Grounding Context

Codebase Context. oh-my-pi는 Pi용 TypeScript 확장 패키지이며 현재 `env-loader`, `workspace-connectors`, `quotio-provider`, `setup-doctor`를 포함합니다. 현재 커넥터 surface는 Linear/Notion의 browser OAuth MCP와 GitHub read-only `gh` CLI 브리지에 초점이 있습니다.

현재 명령은 /connector-login linear|notion, /connector-status [linear|notion], /connector-logout linear|notion, /connector-tools linear|notion이며, 도구는 workspace_mcp_list_tools, workspace_mcp_call_tool, github_gh_cli입니다. `connector-backend-catalog.ts`는 OAuth MCP, CLI, provider backend를 구분하지만 CLI adapter는 지금 GitHub `gh-cli`에 고정되어 있습니다.

Past Learnings. `/connector-login`은 terminal-owning CLI에 위임하지 않고 direct browser OAuth + loopback callback으로 구현해야 한다는 선례가 있습니다. 런타임 tool은 interactive auth를 시작하면 안 되고, 기존 credential을 재사용하거나 setup guidance로 실패해야 합니다. 프로필은 secret-free dry-run intent여야 하며 CWD `.env`와 `ENABLE_*` toggle이 활성화 패턴입니다.

External Context. `gh`와 `glab`는 이미 CLI auth/login/status/logout 패턴이 잘 문서화되어 있고, Atlassian은 API token/OAuth/CLI surface가 있으나 terminal-agent용 Jira+Confluence 통합 패턴은 상대적으로 덜 표준화되어 있습니다. MCP/agent prior art는 interactive setup과 non-interactive tool execution을 분리하는 쪽으로 수렴합니다.

Connector Setup Control Plane

Axis: invocation surfaceConfidence 88%Complexity Medium
Description
하나의 primary surface를 만든다: skill:connector-setup 또는 /connector-setup. 여기서 full/selective/minimal을 선택하고, login/status/logout은 그 안에서 “다음 안전한 human action card”로 노출한다. 기존 전문가용 slash command는 남겨두되, 사용자가 기억해야 할 entrypoint는 setup control plane 하나가 된다.
Basis
direct: 현재 README와 setup doctor는 `/connector-login`, `/connector-status`, `/connector-logout`, `/connector-tools`, profile dry-run, command palette로 setup 동작이 분산되어 있음을 보여준다. external: agent connector prior art는 interactive setup과 non-interactive tool execution을 분리한다.
Rationale
사용자의 요구는 “커넥터 셋업을 skill처럼 쓰고 싶다”는 UX 요구와 “full/selective/minimal”이라는 setup mode 요구가 결합된 것이다. control plane은 이 둘을 한 surface로 묶으면서 runtime tool이 OAuth를 시작하지 않는 기존 안전 원칙을 보존한다.
Downsides
명령/스킬/doctor/profile의 역할 경계가 다시 정의되어야 한다. 처음부터 너무 자동화하면 dry-run·secret-free 원칙을 흔들 수 있어, 초기 버전은 checklist/receipt 중심이어야 한다.
2

Capability Graph + Profile Algebra

Axis: backend/catalog modelConfidence 86%Complexity High
Description
catalog를 service-name list가 아니라 capability graph로 확장한다. `issue-tracker`, `wiki`, `git`, `provider` 같은 capability slot에 tenant(`company`, `personal`), backend kind(OAuth MCP, CLI, provider), auth strategy, setup mode membership, runtime safety policy를 붙인다. profile은 고정 JSON 이름의 조합이 아니라 algebra가 된다: full = company + personal, minimal = base - issue-tracker - wiki - git.
Basis
direct: 현재 catalog는 oauth-mcp | cli | provider backend kind를 이미 구분하지만 WORKSPACE_MCP_SERVICE_IDS와 command parser는 OAuth MCP인 Linear/Notion에만 묶여 있다. GitHub는 별도 gh-cli route helper로 존재한다.
Rationale
Jira/Confluence/GitLab을 Linear/Notion 모양으로 억지 확장하면 명령, setup doctor, profile, prompt snippet이 다시 drift한다. capability graph는 “회사 issue-tracker는 Jira, 개인 issue-tracker는 Linear”처럼 사용자의 언어를 catalog의 언어로 바꾼다.
Downsides
가장 구조적인 변화라 초기 구현 부담이 크다. 기존 profile schema, capability registry, runtime safety ledger와의 호환 계층이 필요할 수 있다.
3

Auth Passport + Scoped Logout

Axis: auth/local-state safetyConfidence 84%Complexity Medium
Description
각 connector의 auth source와 local-state ownership을 “passport”로 보여준다: oauth-store, cwd-env-key, gh-cli-session, glab-cli-session, api-token-guidance, not-configured. `/connector-logout personal:*` 같은 scoped logout은 실행 전 blast radius를 보여준다: 어떤 로컬 OAuth entry를 지우고, 어떤 `.env` key와 CLI session은 건드리지 않는지.
Basis
direct: 현재 auth state는 ~/.pi/agent/workspace-connectors-auth.json에 저장되고, auth file은 0700/0600 권한으로 다뤄진다. 현재 logout은 OAuth state만 지우고 access-key env vars는 그대로 둔다. README와 secret blueprint는 OAuth/cache/session state와 `.env`를 local-only로 분리한다.
Rationale
회사/개인 credential이 한 머신에 공존하면 “로그아웃”은 단순한 기능이 아니라 신뢰 경계다. passport는 status, setup, logout, runtime error가 같은 auth provenance vocabulary를 공유하게 만든다.
Downsides
외부 CLI 세션(`gh`, `glab`)은 Pi가 직접 소유하지 않으므로 passport가 “관찰/가이드”와 “관리/삭제”를 엄격히 구분해야 한다. 이 경계가 흐리면 오히려 사용자가 더 불안해질 수 있다.
4

Readiness-Gated Tool Affordances

Axis: diagnostics/tool exposureConfidence 78%Complexity High
Description
tool exposure를 “extension이 켜졌으면 무조건 등록”에서 “setup receipt와 readiness gate가 허용한 affordance만 노출”로 바꾼다. 최소 setup은 issue/wiki/git 도구를 실제로 숨기고, selective/full은 connector별로 read-only, confirm-write, not-configured, hidden-by-mode 같은 상태를 tool card로 제공한다.
Basis
direct: 현재 workspace_mcp_list_tools, workspace_mcp_call_tool, github_gh_cli가 LLM-callable tool로 등록되고, MCP call tool은 exact schema와 destructive write confirmation을 요구한다. setup doctor는 connector catalog, runtime safety ledger, gh auth를 이미 요약한다.
Rationale
커넥터 수가 늘어나면 “tool은 보이는데 auth가 안 됨” 또는 “minimal인데 agent가 GitLab을 제안함” 같은 UX debt가 커진다. readiness-gated affordance는 tool visibility 자체를 setup/safety의 산출물로 만든다.
Downsides
Pi extension API의 active tools 관리와 prompt snippets 갱신 타이밍을 신중히 다뤄야 한다. 너무 많이 숨기면 assistant가 setup guidance조차 못 찾을 수 있어, hidden과 setup-hint의 균형이 필요하다.
5

Atlassian/GitLab Staged Entry

Axis: setup profilesConfidence 80%Complexity Medium
Description
회사 stack을 한 번에 완성하려 하지 말고 단계적으로 들인다. GitLab은 이미 인증된 glab를 전제로 GitHub `gh` 패턴을 복제한 read-only CLI backend로 시작한다. Jira/Confluence는 catalog와 setup/profile에는 보이게 하되, 안전한 auth route(API token env, OAuth, CLI/API bridge)가 선택될 때까지 runtime tool은 gated 상태로 둔다.
Basis
direct: 현재 GitHub는 backendKind: "cli", adapterKind: "gh-cli", read-only github_gh_cli로 존재한다. external: GitLab `glab` auth/status는 잘 문서화되어 있고, Atlassian은 API token/OAuth/CLI surface가 있으나 terminal-agent pattern은 덜 표준화되어 있다.
Rationale
가장 확실한 회사 connector value는 GitLab CLI에서 즉시 얻고, 가장 불확실한 Jira/Confluence runtime auth는 setup-visible/runtime-gated로 안전하게 탐색한다. 이렇게 하면 회사 stack 요구를 충족하면서도 Atlassian 쪽에서 성급한 auth 설계를 하지 않는다.
Downsides
Atlassian 사용자는 “왜 setup에는 보이는데 도구는 아직 안 되나”를 궁금해할 수 있다. staged 상태의 문구와 next action이 매우 명확해야 한다.
