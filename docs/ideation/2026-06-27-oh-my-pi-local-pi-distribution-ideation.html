아이디어 탐색: oh-my-pi를 내 로컬 Pi 배포판으로 키우기

지금의 oh-my-pi는 개인용 Pi 확장 모음에 가깝다. 이걸 새 노트북이나 다른 프로젝트에서도 바로 되살릴 수 있는 “내 로컬 Pi 배포판”으로 만들려면 어디부터 손대야 할지 정리했다. 핵심은 설치 프로필, 커넥터별 backend 선택(CLI/MCP/API/provider), Quotio·CLI proxy provider, 안전한 시크릿 경계, 그리고 처음 실행할 때 막히지 않는 셋업 경험이다.

Grounding Context

Codebase Context — 지금 코드베이스에서 보이는 것

현재 형태

package.json은 이미 env-loader, quotio-provider, workspace-connectors를 Pi 패키지로 등록하고 있다. README도 “여러 머신에서 공유하는 개인 Pi 패키지”를 목표로 잡고 있다. 다만 지금은 재현 가능한 셋업이라기보다 예시 파일과 README를 보고 손으로 맞추는 단계에 가깝다.

설정 모델

현재 설정의 중심은 Pi를 실행한 디렉터리의 .env다. 이 파일 하나가 확장 토글과 실제 값을 함께 담고, 기존 환경변수보다 우선한다. 이 방식은 단순하고 프로젝트별로 격리하기 좋지만, 새 확장을 추가할 때마다 KNOWN_TOGGLES 같은 중앙 목록을 손봐야 한다.

커넥터와 Provider

Linear와 Notion은 remote MCP URL을 직접 코드에 넣어두었고, 매번 mcp-remote@latest를 npx로 실행한다. GitHub는 로컬 gh auth 세션을 읽기 위주로 재사용한다. 반면 Quotio는 현재 로컬 CLI가 없어서(quotio: command not found) env 기반 provider 경로가 필요하다. 즉 커넥터마다 CLI, MCP, API/provider 중 어떤 backend를 쓸지 판단하는 구조가 더 맞다.

외부 패턴

Claude Code와 Codex/OpenAI 쪽 흐름은 remote MCP, OAuth, scope가 있는 설정, 승인/allowlist 쪽으로 모이고 있다. Dev Containers와 Home Manager는 “프로필과 lockfile로 환경을 되살린다”는 관점을, LiteLLM/Ollama는 OpenAI-compatible proxy를 표준 어댑터처럼 쓰는 관점을 준다.

Topic Axes

Topic Axes — 어디를 봐야 하나

1. Package profiles단순 설치 목록을 넘어, 프로필·lock·receipt·bundle로 다시 만들 수 있는 구성을 만든다.
2. Connector backend strategy각 커넥터가 CLI, MCP, API/provider 중 어떤 backend를 쓸지 선언하고 선택하게 만든다.
3. CLI proxy providersQuotio와 CLIProxyServer류 provider를 매번 새로 짜지 않고, 반복 가능한 contract로 다룬다.
4. Config, secrets, diagnosticsCWD `.env`, OAuth state, provider secret, setup 상태를 한눈에 이해하고 고칠 수 있게 만든다.
5. Trust and modularityopt-in, approval, policy, extension boundary를 README 경고가 아니라 실제 런타임 구조로 만든다.

1

Idea 1. 실행 가능한 프로필 팩 + Lockfile

Conf 88%Cx Medium

Description: default, workspace, proxy-provider, full 같은 프로필을 “설치할 패키지 목록”이 아니라 실제로 실행 가능한 workspace contract로 정의한다. 프로필 하나 안에 Pi package ref, git commit/tag, extension capsule, 켤 토글, 필요한 secret reference, prompts/themes, connector/provider 기본값, 생성된 lock/receipt까지 담는다.

Illustrative: 프로필은 읽고 따라 하는 문서가 아니라, 적용할 수 있는 구성 단위가 된다.

Basis

direct: settings.example.json:2-5는 아직 버전이 고정되지 않은 package list만 담고 있고, README의 설치 명령도 git source string만 보여준다. 반면 package.json:20-27은 extensions/prompts/themes를 이미 manifest로 등록한다. external: Pi package의 manifest/filtering, 그리고 Dev Containers/Home Manager의 profile/lock 관점이 이 문제와 잘 맞는다.

Axis
Package profiles and reproducible installation
Rationale
새 머신 셋업, 프로젝트별 구성, 패키지 업데이트 검증을 모두 같은 profile contract 위에서 다룰 수 있다.
Downsides
초기에 schema와 lock/receipt 생성 규칙을 정해야 한다. Pi의 기존 settings 포맷과도 어긋나지 않게 맞춰야 한다.
2

Idea 2. Setup Doctor + Command Palette

Conf 91%Cx Medium

Description: /oh-my-pi-doctor와 선택형 /oh-my-pi command palette를 만든다. Doctor는 CWD `.env`, 켜진/꺼진 토글, 필요한 secret, Quotio 연결 상태, connector OAuth 상태, gh auth, package/profile lock drift, 누락된 prompts/themes, 커밋되면 안 되는 파일까지 한 번에 확인하고 다음에 뭘 하면 되는지 알려준다.

Basis

direct: 지금은 env-loader의 loaded/disabled 알림, /quotio-status, /connector-login//connector-tools가 각각 따로 진단을 해준다. 근거 수집에서도 generated `.env.example`, setup doctor, 제대로 된 테스트 스크립트가 없다는 점이 드러났다. reasoned: opt-in 구조는 안전하지만, 사용자는 “왜 안 켜졌는지”를 한 화면에서 바로 알아야 한다.

Axis
Configuration, secrets, toggles, and diagnostics
Rationale
설치 직후 “분명 설치했는데 왜 아무 도구도 안 보이지?”라는 답답함을 가장 빨리 줄인다.
Downsides
각 capability가 health probe를 제공해야 한다. 그래서 registry/capsule과 함께 설계해야 중복을 줄일 수 있다.
3

Idea 3. 자기 설명형 Capability Registry + Extension Capsules

Conf 86%Cx Medium

Description: extension/provider/connector를 각각 capsule로 보고, capsule manifest에 toggle, env schema, commands/tools, provider registration, connector URL, profile tag, load order, safety class, diagnostics hook을 선언한다. 그러면 env-loader, doctor, README table, `.env.example`, profile compiler가 모두 같은 registry를 읽을 수 있다.

Basis

direct: extensions/env-loader/index.ts:5-8의 KNOWN_TOGGLES는 지금 중앙에 하드코딩돼 있다. AGENTS.md:24-30은 extension 작성 패턴을 이미 정리해두었고, package.json:20-27은 리소스 등록 지점을 제공한다. reasoned: connector/provider가 늘어날수록 docs, loader, setup snippet, doctor가 서로 어긋날 가능성이 커진다.

Axis
Trust, safety, modularity, and extension boundaries
Rationale
oh-my-pi가 3개 extension짜리 모음에서 “개인 배포판”으로 커질 때 중심이 될 컴포지션 계층이다.
Downsides
manifest가 너무 무거우면 작은 extension 하나 추가하는 일도 부담스러워진다. 필수 필드와 optional hook을 잘 나눠야 한다.
4

Idea 4. Connector Backend Catalog + Adapter Router

Conf 84%Cx Medium/High

Description: 커넥터를 “MCP로 붙일 것인가, CLI로 붙일 것인가”로 미리 고정하지 않는다. 각 connector catalog entry가 지원 backend(cli, mcp, http-api, provider), 기본 backend, auth/status check, safety class, profile override를 선언한다. Adapter router는 현재 머신에서 가능한 backend를 고르고, 실패하면 명확한 fallback 또는 수동 조치를 알려준다.

Illustrative: 새 서비스를 붙일 때 “CLI냐 MCP냐”를 코드에서 싸우지 않고, catalog entry와 adapter가 결정한다.

Basis

direct: workspace-connectors/index.ts:7-16은 Linear/Notion URL을 직접 들고 있고, GitHub는 이미 gh CLI 세션을 재사용한다. 사용자가 확인한 quotio status는 command not found였기 때문에 Quotio는 CLI-first가 아니라 env/provider backend가 필요하다. external: Claude/OpenAI 쪽 connector 모델은 remote MCP, OAuth, scoped config, allowlist를 중심으로 하지만, 로컬 CLI auth를 재사용하는 도구도 같은 connector 표면에서 다룰 수 있다.

Axis
Connector backend strategy and UX
Rationale
GitHub처럼 CLI가 좋은 서비스, Linear/Notion처럼 MCP가 나을 수 있는 서비스, Quotio처럼 env/provider가 필요한 서비스를 같은 connector 표면에서 다룰 수 있다.
Downsides
backend별 auth/status 명령을 표준화해야 한다. MCP adapter를 쓴다면 별도 패키지로 분리하고, CLI가 없는 provider는 local config overlay에 의존해야 한다.

Idea 5. Provider Adapter Kit + Capability Contract

Conf 82%Cx Medium

Description: Quotio provider에서 시작해, OpenAI-compatible proxy/provider를 붙이는 공통 adapter kit를 만든다. Adapter contract는 base URL, auth resolver, model discovery, timeout/auth diagnostics, capability/cost overrides, model aliases, cache TTL, default model roles를 정의한다.

Basis

direct: quotio-provider/index.ts:18-31은 /models를 bearer auth로 가져오고, index.ts:34-45는 모델 ID에 들어간 단어로 vision/reasoning/context/cost를 추정한다. index.ts:90-130에는 connectivity/auth/status check가 이미 있다. external: LiteLLM/Ollama 계열도 OpenAI-compatible facade를 중심으로 local/proxy model routing을 다룬다.

Axis
CLI proxy and model provider integration
Rationale
Quotio뿐 아니라 CLIProxyServer류 provider를 붙일 때도 discovery/status/capability 로직을 다시 만들 필요가 없다.
Downsides
provider마다 model metadata source가 다를 수 있다. override 파일과 remote discovery 중 무엇을 우선할지 정해야 한다.

Idea 6. 비밀값 없는 Blueprint + Secret References

Conf 78%Cx Medium/High

Description: “로컬 설정은 전부 커밋하면 안 된다”는 전제를 조금 나눈다. 비밀이 아닌 blueprint는 버전 관리하고, 비밀값만 local overlay로 둔다. 예를 들어 profiles/*.json, .env.schema, connector catalog, safety policy는 커밋하고, 실제 값은 .env.local, OAuth state, keychain/1Password/pass 같은 secret reference resolver에서 가져온다.

Basis

direct: AGENTS.md:34-37은 CWD `.env` override와 env-loader 선행 로드를 정의하고, README.md:48-51과 .gitignore:5-9는 auth/session/.mcp-auth/.env를 local-only로 둔다. reasoned: 다른 머신에서 되살려야 하는 것은 secret 값 자체가 아니라, 어떤 capability를 켜고 어떤 resolver 경계를 쓸지에 대한 의도다.

Axis
Configuration, secrets, toggles, and diagnostics
Rationale
안전하게 공유하면서도 재현성을 얻는다. 코드 리뷰에서는 “무엇이 켜지는지”는 보이지만 token 값은 보이지 않는다.
Downsides
secret resolver를 너무 크게 만들면 로컬 setup이 오히려 복잡해질 수 있다. 초기 버전은 `.env.schema`와 optional env/keychain resolver 정도로 작게 시작하는 편이 좋다.
7

Idea 7. Runtime Safety Policy Ledger

Conf 80%Cx Medium

Description: README 경고와 prompt guideline에만 기대지 않고, runtime policy ledger를 둔다. Policy는 connector/provider/tool별 read-only, confirm-write, trusted-project, allow/deny list, approval expiry, redacted audit trail을 선언하고, wrapper는 실제 tool을 실행하기 전에 이 정책을 확인한다.

Illustrative: 안전 규칙은 모델이 기억해야 하는 약속이 아니라, tool boundary에서 확인하는 정책이 된다.

Basis

direct: workspace_mcp_call_tool에는 destructive write 전에 확인하라는 prompt guideline이 있고, github_gh_cli는 create/edit/delete/merge 같은 verb blocklist로 위험한 명령을 막는다. external: Claude/OpenAI-style remote tool 사용은 approval/allowlist와 trust boundary를 중요하게 다룬다.

Axis
Trust, safety, modularity, and extension boundaries
Rationale
강력한 connector를 켤수록 안전 정책은 README가 아니라 실행 경계에서 다뤄져야 한다.
Downsides
confirmation이 너무 많으면 UX가 나빠진다. read-only default와 scoped escalation 사이의 균형을 잡아야 한다.
Rejection Summary

Rejection Summary — 제외한 아이디어

# Idea Reason Rejected
1 Fresh-machine bootstrap wizard 별도 아이디어로 두기보다는 Idea 1의 Profile Packs와 Idea 2의 Doctor에 들어가는 onboarding flow로 보는 편이 더 낫다.
2 Connector mini-app / OAuth account switchboard connector status UX는 Idea 4의 backend catalog에 포함된다. 다중 OAuth 계정 switching은 backend별 auth 상태를 먼저 표준화한 뒤 다루는 편이 낫다.
3 Encrypted local state migration kit 흥미롭지만 Pi auth, sessions, CLI auth까지 다루면 oh-my-pi가 책임져야 할 범위를 쉽게 넘어선다.
4 Project-local Pi lens CWD `.env` override를 잘 확장하는 방향이지만, profile/blueprint 모델 안의 하위 개념으로 다루는 편이 낫다.
5 Extension scaffold + smoke tests 유용한 후속 DX 작업이다. 다만 먼저 capability registry가 있어야 scaffold가 무엇을 생성할지 안정된다.
6 Safe export/import boundary bundle secret/auth/session을 제외하는 원칙은 중요하지만, Idea 1의 lock/receipt와 Idea 6의 blueprint에 자연스럽게 포함된다.
7 OAuth-first Quotio login 현재 근거는 bearer API key 방식뿐이다. Quotio가 OAuth를 지원하는지 확인되지 않아 우선순위에서 제외한다.
8 Connector/tool schema cache 독립 방향으로 두기보다는 Idea 4의 MCP adapter backend가 제공해야 할 세부 기능으로 포함한다.
