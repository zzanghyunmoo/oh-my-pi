결론부터: 지금의 projects/oh-my-pi/만 설치해서는 현재 pi list에 보이는 전체 패키지와 그 패키지들이 제공하는 스킬을 재현할 수 없다. oh-my-pi는 자기 확장과 설정 보조 도구를 제공하지만, pi-subagents, pi-lens, compound-engineering-plugin 같은 외부 Pi 패키지를 자동 설치하거나 번들링하지 않는다. 다만 목표 UX는 사용자가 원본 패키지명을 기억하지 않고 모든 익스텐션과 스킬을 omp: <스킬 명> 형태로 활용하는 것이다.

근거 맥락

현재 사용자 패키지 상태

~/.pi/agent/settings.json 기준 현재 패키지는 다음 11개다.

npm:pi-subagents, npm:pi-web-access, npm:pi-lens, npm:pi-powerline-footer, npm:pi-ask-user, git:github.com/EveryInc/compound-engineering-plugin, npm:pi-linear, npm:pi-notion, npm:pi-github, npm:pi-clear, git:github.com/zzanghyunmoo/oh-my-pi.

oh-my-pi 패키지 상태

package.json은 env-loader, quotio-provider, workspace-connectors, setup-doctor 확장과 prompts/themes만 노출한다. pi.skills는 없고, 외부 Pi 패키지를 의존성으로 설치하지도 않는다.

프로필 상태

docs/profiles/full.profile.json은 oh-my-pi, pi-web-access, @leing2021/super-pi, pi-clear만 기록한다. 반면 현재 설치된 pi-lens, pi-subagents, pi-ask-user, compound-engineering-plugin 등은 빠져 있다.

Pi 패키지 규칙

Pi 패키지는 자기 pi manifest나 convention directory로 리소스를 노출한다. 일반 npm dependency를 설치한다고 그 dependency의 Pi 스킬/확장이 자동으로 로드되지는 않는다. 다른 Pi 패키지 리소스를 함께 싣고 싶다면 dependency/bundledDependencies와 node_modules/... 경로를 manifest에 명시해야 한다.

주제 축

  • Manifest 정확도 — 어떤 패키지 ref가 필요한지, 실제 리소스 커버리지와 프로필이 일치하는지.
  • 스킬·도구 커버리지 — 스킬, 도구, 커맨드, provider, prompt, theme가 어느 패키지에서 오는지.
  • 안전한 setup UX — secret-free, OAuth/local-state-free, 기본 비파괴 흐름을 유지하는지.
  • OMP 사용자 네임스페이스 — 실제 제공 패키지가 무엇이든 사용자는 omp: <스킬 명>으로 발견하고 실행할 수 있는지.
  • 업데이트 거버넌스 — npm/git ref, pin, update lane이 명시되어 새 머신과 기존 머신이 같은 방향으로 진화하는지.
  • 진단과 drift 감지 — 현재 머신과 선택 프로필의 차이를 사용자가 즉시 이해할 수 있는지.

이번 반영 범위

이 문서는 실행 계획이 아니라 ideation artifact다. 따라서 여기서는 “무엇을 재현해야 하는가”와 “어떤 순서로 요구사항화할 것인가”만 고정하고, README·plan·코드 수정은 후속 requirements/plan 단계로 넘긴다.

반영 원칙: 자동 설치나 기존 문서 변경보다 먼저, 현재 패키지·스킬 상태와 프로필이 어긋나는 지점을 이 문서 안에서 명확히 남긴다. 이후 작업은 이 ideation을 근거로 별도 문서에서 진행한다.

Profile Pack을 canonical distribution manifest로 승격

축
Manifest 정확도
설명
프로필 팩을 단순한 setup 의도 기록에서 “로컬 Pi 배포판의 기준 manifest”로 승격한다. full을 갱신하거나 local-full 같은 새 프로필을 만들어 현재 필요한 모든 package ref를 넣고, lock receipt로 기대 상태를 고정한다.
근거
direct: 현재 settings에는 11개 패키지가 있지만 full.profile.json에는 4개 ref만 있다. 동시에 profile:verify는 프로필과 lock을 deterministic하고 secret-free하게 검증할 수 있음을 이미 보여준다.
왜 중요한가
“새 머신에 무엇을 설치해야 하는가?”라는 질문이 기억이나 특정 머신의 settings가 아니라 커밋된 파일에서 답해진다.
단점
모든 패키지를 full에 넣을지, dev/CE/workspace처럼 프로필을 더 쪼갤지 먼저 정해야 한다.

I2.

Doctor에 현재-vs-프로필 drift 리포트 추가

축
진단과 drift 감지
설명
/oh-my-pi-doctor가 선택한 프로필 lock과 현재 설치된 package/skill 상태를 비교하게 한다. 누락, 초과, 비활성, stale 항목을 보여주면 “설치된 것 같은데 왜 스킬이 안 보이지?” 같은 혼란을 바로 줄일 수 있다.
근거
direct: setup-doctor는 이미 env, capability registry, connector catalog, runtime safety policy, Quotio 연결성, gh auth, local-only path를 요약한다. package/profile drift는 같은 진단 표면에 자연스럽게 들어간다.
왜 중요한가
설치 자동화를 당장 만들지 않아도, 사용자는 현재 머신이 프로필과 같은 상태인지 Pi 안에서 확인할 수 있다.
단점
Pi settings와 package list를 읽는 경로를 안전하게 정해야 하고, local path를 불필요하게 노출하지 않도록 다듬어야 한다.

I3.

profile:import-current 스냅샷 명령

축
안전한 setup UX
설명
현재 설치된 package ref와 가능한 리소스 표면을 읽어 draft profile이나 patch를 만들어 주는 명령을 추가한다. 자동 설치나 자동 커밋은 하지 않고, 리뷰 가능한 migration artifact만 생성한다.
근거
reasoned: 이번 간극도 pi list, settings, package.json, profile JSON을 수동 비교해서 드러났다. 스냅샷 명령은 이 수동 감사를 반복 가능한 유지보수 흐름으로 바꾼다.
왜 중요한가
앞으로 새 Pi 패키지를 설치할 때마다 프로필은 다시 drift될 수 있다. import 명령이 있으면 실제 사용 상태를 안전하게 프로필 후보로 끌어올릴 수 있다.
단점
CLI 출력 파싱은 깨지기 쉽다. settings JSON을 직접 읽는 쪽이 안정적이지만 project-local package까지 포괄하려면 추가 설계가 필요하다.

확인형 bootstrap apply

축
안전한 setup UX
설명
profile:apply의 기본값은 계속 dry-run으로 유지하되, --execute 같은 명시적 옵션이나 Pi 커맨드를 통해 사용자가 확인한 뒤에만 누락된 pi install을 실행하게 한다.
근거
direct: 현재 스크립트는 install intent를 출력하면서 pi install, .env 작성, OAuth 시작을 하지 않는다고 명시한다. Pi package docs는 외부 패키지를 각 install spec으로 명시 설치하는 모델을 제공한다.
왜 중요한가
Manifest는 재현성을 문서화하고, 확인형 bootstrap은 재현성을 실사용 가능하게 만든다. 안전한 기본값과 편의성을 동시에 잡을 수 있다.
단점
부분 실패, user/project scope, 기존 설치 충돌, 신뢰/보안 메시지까지 다뤄야 하므로 구현 부담은 가장 크다.

패키지·스킬 커버리지 매트릭스

축
스킬·도구 커버리지
설명
각 프로필 package가 제공하는 extension, command, tool, provider, skill, prompt, theme를 표로 생성한다. 각 패키지가 왜 필요한지, required인지 optional인지까지 함께 보여준다.
근거
direct: 현재 보이는 주요 스킬은 pi-subagents, pi-web-access, pi-lens, pi-ask-user, compound-engineering-plugin 같은 외부 패키지에서 온다. oh-my-pi 자체는 pi.skills를 선언하지 않는다.
왜 중요한가
사용자의 질문은 package와 skill을 함께 묻고 있다. 커버리지 표가 있으면 “패키지는 설치됨”과 “스킬이 실제로 로드됨”을 구분할 수 있다.
단점
단순 파일 검색은 test fixture나 disabled resource를 잘못 포함할 수 있다. Pi의 manifest/filter 규칙에 맞춰 discovery해야 한다.

Pin/Update 정책 lane

축
업데이트 거버넌스
설명
프로필에 package별 update 정책을 둔다. 예를 들어 latest npm 추적, exact npm version pin, git branch 추적, git commit/tag pin을 구분하고, lock receipt에는 pi update --extensions가 어떤 항목을 움직일 수 있는지 표시한다.
근거
direct: Pi package docs는 npm versioned spec, git ref, update reconciliation의 차이를 설명한다. 현재 setup은 npm package와 git package를 모두 포함한다.
왜 중요한가
로컬 배포판은 “오늘 새 머신을 재현한다”뿐 아니라 “내일 어떻게 업데이트되는가”까지 포함한다. update lane이 있어야 재현성과 최신성이 충돌하지 않는다.
단점
빠르게 변하는 도구 패키지에서 편의성과 고정성의 균형을 정해야 한다.

omp: <스킬 명> 사용자 네임스페이스

축
OMP 사용자 네임스페이스
설명
실제 스킬·익스텐션 제공자가 pi-lens, pi-subagents, compound-engineering-plugin 같은 외부 패키지여도, 사용자는 모두 omp: <스킬 명> 형태로 발견하고 실행한다. OMP는 원본 package ref와 실제 skill/command/provider를 매핑하는 facade 역할을 맡는다.
근거
reasoned: 현재 문제는 단순 설치 gap뿐 아니라 “어떤 패키지가 어떤 스킬을 제공하는지 사용자가 기억해야 한다”는 UX gap이다. oh-my-pi가 로컬 배포판이라면 사용자 표면은 패키지명이 아니라 OMP 네임스페이스로 통일되어야 한다.
왜 중요한가
새 머신이나 새 세션에서도 사용자는 omp: ce-plan, omp: lsp-navigation, omp: ask-user처럼 한 가지 규칙만 기억하면 된다. 내부 구현은 upstream 패키지에 위임하되, 사용자 경험은 oh-my-pi가 소유한다.
단점
실제 Pi 런타임이 지원하는 skill naming, prompt alias, command alias 규칙을 확인해야 한다. upstream 이름을 숨기면 문제 해결 시 출처를 놓칠 수 있으므로 doctor와 coverage matrix에는 원본 package mapping을 함께 보여줘야 한다.

권장 반영 순서

일곱 아이디어는 독립 목록이 아니라 의존성을 가진 흐름으로 다룬다. 핵심은 먼저 재현 대상·커버리지·OMP 사용자 네임스페이스를 확정하고, 그 다음 진단과 자동화 UX를 얹는 것이다.

순서 아이디어 반영 내용 이유
1 I1 + I5 + I7 Canonical profile manifest, 패키지·스킬 커버리지 매트릭스, omp: <스킬 명> 사용자 네임스페이스 어떤 package ref와 skill/tool 표면을 재현해야 하는지, 그리고 사용자가 어떤 OMP 이름으로 쓸지를 먼저 고정해야 doctor와 apply가 같은 기준을 쓸 수 있다.
2 I2 현재-vs-프로필 drift 리포트 설치 자동화 전에도 사용자가 현재 머신의 누락·초과·비활성 상태를 이해할 수 있다.
3 I3 profile:import-current draft artifact 실제 사용 상태를 안전하게 프로필 후보로 끌어올려, future drift 대응을 반복 가능하게 만든다.
4 I4 확인형 bootstrap apply dry-run 기본 계약을 유지하면서, 사용자가 명시적으로 확인한 경우만 누락 package 설치를 실행한다.
5 I6 Pin/update lane 재현 가능한 설치가 안정화된 뒤, npm/git ref가 앞으로 어떻게 움직일지 정책화한다.

요약: I1 + I5 + I7 → I2 → I3 → I4 → I6. “바로 설치”보다 “무엇이 빠졌는지, 사용자가 어떤 omp: 이름으로 쓸지 안전하게 보이게 만들기”가 먼저다.

탈락 요약

# 아이디어 탈락 이유
1 모든 외부 Pi 패키지를 vendoring하는 meta-package Pi 패키지 규칙상 가능성은 있지만 결합도와 업데이트 부담이 크다. 먼저 profile/doctor 기반으로 간극을 줄이는 편이 효율적이다.
2 외부 패키지를 oh-my-pi 내부에서 재구현 주제 대체에 가깝고 비용이 과하다. 목표는 upstream 기능 소유가 아니라 로컬 setup 재현성과 omp: 사용자-facing alias다.
3 OAuth/session/auth 상태까지 커밋해 원클릭 복원 프로젝트 가드레일과 secret/OAuth boundary를 정면으로 위반한다.
4 profile:apply가 조용히 모든 패키지를 설치 현재의 비파괴 안전 계약과 맞지 않는다. 실행 모드는 반드시 명시적이고 확인형이어야 한다.
5 CI가 한 사용자의 settings와 정확히 일치해야 통과 너무 machine-specific하다. CI 실패 조건보다는 doctor drift 리포트나 fixture 기반 검증으로 다루는 편이 낫다.
6 README 체크리스트만 갱신 도움은 되지만 단독으로는 기대 효과가 낮다. 프로필/lock/doctor에서 생성되거나 그와 함께 제공되어야 한다.
